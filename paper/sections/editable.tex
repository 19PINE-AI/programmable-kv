\section{Consequence I: the cache is editable}
\label{sec:editable}

\Cref{fig:ops} contrasts the two cache operations this section and the next make precise: editing appends
a salient correction; composing repositions and splices precompiled KV.

\paragraph{Naive editing fails---by design.} The mechanism predicts the in-place edit will fail, and it
does: refreshing the field's KV while reusing the stale downstream leaves the decision at the old value
(\Cref{fig:teaser}b). The prefix before the field is genuinely reusable (deviation $0.0$); the problem is
that the conclusion was already memoized after it.

\begin{figure}[t]
\centering
\includegraphics[width=\linewidth]{fig_operations}
\caption{\textbf{Two cache operations.} (a) \emph{Editable}: append a salient erratum that amends the
stale notes ($O(1)$ extra work, the whole prefix is reused). (b) \emph{Composable}: precompute a skill's
KV in isolation, RoPE-reposition it to the target positions, and splice it in ($O(L)$, no reprefill).}
\label{fig:ops}
\end{figure}

\paragraph{Two fixes, neither a full reprefill.} If the stale notes carry an old conclusion, there are
two correct interventions. \emph{(i) Recompute the affected notes}: refresh the field plus the few
highest-effect downstream tokens that memoized the old conclusion---\fieldsel{}@$K$, recomputing only the
dependent suffix rather than the whole context. \emph{(ii) Append an erratum}: one salient line late in the
context---``\texttt{[STATE UPDATE] field $\rightarrow$ new; overrides any earlier value and
conclusion}''---so the decision token attends to a fresh, authoritative note (verbatim template in
\Cref{app:examples}). The erratum is the robust default: appended after the field (\texttt{field+erratum})
it matches the strong \emph{hoist-to-end} oracle \emph{without rewriting the prompt} ($0.97$ under
chain-of-thought on Qwen3-8B, \Cref{fig:overview}a), and being append-only it composes with prefix caching
(\Cref{sec:systems}).

\paragraph{Reasoning vs.\ instruction-tuned: a stark split.} Which fix you \emph{need} depends sharply on
the model class. On a \emph{reasoning} model a near-free third option appears---refresh the field's own KV
alone (${\sim}1\%$ compute) and nothing else---because the chain re-reads the refreshed field and
re-derives the conclusion instead of trusting the stale note; field-only recovery is $1.00$ on Qwen3-8B.
On an \emph{instruction-tuned} model the identical edit is \emph{ignored} ($0.00$): with no chain to
re-derive, the decision commits to the memoized note (\Cref{fig:overview}c). Instruction-tuned models
therefore require one of the two real fixes above. The mechanics are simple---an in-place edit only helps
if some later computation actually re-reads the field---and we keep them brief here; the per-model
stickiness, the \fieldsel{}@$K$ sweep ($K^\star\approx 4$ at $8$B but $>64$ at $4$B, varying
non-monotonically with scale; \Cref{fig:edit}a,b), and the layer-wise account are detailed in
\Cref{app:edit,app:deepmech}. We report \fieldsel{}@$K$ honestly as a genuine but \emph{unreliable}
surgical tool, not a default.

\paragraph{A frontier, not a winner.} A controlled comparison (full table in \Cref{app:edit}) shows no single
dominant method. Hoist-to-end is cheapest but demands prompt surgery and pre-identification of every field;
\texttt{field+erratum} matches it with no surgery at a one-line append; \inplace{} is near-free but only
under reasoning; a KV-deviation-ranked selective recompute (CacheBlend-style) underperforms here because it
chases changed keys rather than the tokens that \emph{memoized the conclusion}. The practical
recommendation is \texttt{field+erratum} as the robust default, with \inplace{} as a free fast-path for
reasoning models.

\paragraph{Why not edit the weights?} A natural objection: to act on a changed field, why not edit
the model's \emph{weights} (ROME/MEMIT) or fine-tune, rather than the cache? We compare on the paper's
gated task (status \texttt{pending}$\rightarrow$\texttt{shipped}, so \emph{cancel}$\rightarrow$\emph{deny})
against a faithful rank-one ROME---validated on the canonical factual edit first (``the Eiffel Tower is
in'' \emph{Paris}$\rightarrow$\emph{Rome}, locality intact), so the baseline is not crippled---and a LoRA
fine-tune (\Cref{tab:weight}; methodology in \Cref{app:systems}). All of \fielderr{}, ROME, and LoRA \emph{succeed} at flipping the target
decision. But a weight edit is \emph{global}: the same model instance can no longer hold
\texttt{status=shipped} for one request and \texttt{pending} for another, so \emph{all} concurrent orders
that are genuinely still pending are wrongly flipped (cross-request contamination $1.0$), and half of an
unrelated decision battery drifts ($0.5$)---the ROME/fine-tune specificity tax---at $3$--$6$\,s per edit
(plus a one-time covariance pass for ROME). The append-only erratum lives in a \emph{per-sequence} cache:
zero cross-request contamination, zero collateral, $114$\,ms, and it composes with prefix caching
(\Cref{sec:systems}). Weight editing targets \emph{durable, global facts}; mutable per-request, per-turn
state is the wrong job for it---which is precisely the niche the editable KV cache fills.
